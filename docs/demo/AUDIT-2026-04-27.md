## Demo-killer findings

### 1. Five "Try" CTAs on the portal landing → red error pages
**Where it bites:** the marketing landing has these `portal-card__try` hrefs hard-coded:

```
/admin/composability
/distributor/approvals
/investor/products
/issuer/products
/ops/audit
```

Every one of those routes calls the BFF and renders `Could not load products.` (or equivalent) when the call fails. The interviewer's most natural first click is "Try", and it lands on a broken page before they see anything good.

**Why it's broken:** two compounding causes.
- `hydrax-portals` has no `VITE_BFF_URL` set (verified: `railway variables --service hydrax-portals` returns zero matches for `VITE_BFF`). The api-client falls through to `http://localhost:8080` per [web/packages/api-client](../../web/packages/api-client/) defaults.
- `hydrax-bff` service exists but has never had a successful deployment (`railway status --json` shows `status=?`, `updated=never`).

So even if `VITE_BFF_URL` were set, there would be no backend to hit yet.

### 2. The Health pages don't actually prove health
`/investor/health` and `/ops/health` are wired to poll BFF `/healthz/composite` every 5s. With the bundle pointed at localhost, the page shows "all upstreams down" forever. That's the *one* page where the interviewer is *expecting* a status display — and it lies.

## Two fix paths — pick one

### Path A — Hide the broken paths (safe, ~15 min, demo-eve safe)

Make the 5 "Try" CTAs land on portal home routes (which render hero + EmptyState and look polished) instead of BFF-backed routes (which render the red error). Single static-site change to one file in `hydrax-portals`. No backend touched.

**Concrete change:** in the deployed portals landing HTML, rewrite:
```
/admin/composability   → /admin/
/distributor/approvals → /distributor/
/investor/products     → /investor/
/issuer/products       → /issuer/
/ops/audit             → /ops/
```

Each home route renders a portal-specific hero JPEG + EmptyState. Looks intentional. The interviewer sees five distinct branded surfaces and forms a positive impression. They never hit the error state unless they manually type a sub-route, which they won't.

**Trade-off:** the demo is purely visual. If the interviewer asks "can I see actual data?", the answer is "the workflow service is running ([health endpoint to prove]); the UI is wired but the BFF deployment is the next ship — you can see the test suite proving the wiring at [link]". Honest and consistent with the documented "deferred-not-resolved" framing already in CLAUDE.md.

**Risk:** essentially zero. Single file change to a static site.

### Path B — Deploy BFF + rebuild portals with correct env (~45-90 min, demo-eve risky)

Three steps, in order:
1. `railway up --service hydrax-bff` from the bff workspace root, wait for green, verify `https://hydrax-bff-production.up.railway.app/healthz` returns 200.
2. `railway variables --service hydrax-portals --set VITE_BFF_URL=https://hydrax-bff-production.up.railway.app`.
3. `railway up --service hydrax-portals` to rebuild the SPAs with the env baked into the bundle (`import.meta.env.VITE_BFF_URL` is a build-time substitution, not runtime).
4. CORS sanity check: `BFF_CORS_ALLOWED_ORIGIN` is already set to `https://hydrax-portals-production.up.railway.app` on the bff service, which matches.
5. Verify by hitting `/issuer/products` — should now load (empty list, but no red alert) and the Network tab should show a real call to `hydrax-bff-production.up.railway.app`.

**Trade-off:** real, working data layer. Massive credibility uplift.

**Risk:** non-trivial. The BFF has never been deployed in this project; the first build can fail (Dockerfile, Nixpacks, env, port mismatch). Two deploys + a build mid-pipeline. If anything breaks at 11 PM and there's no time to roll back, you ship in worse shape than today. Recommend doing this **only if there's at least 2 hours of buffer before the demo**.

## Safe demo flow if no fixes ship (Path Z — zero risk)

Use this exact click-path. Every URL has been verified 200. None of these touch the BFF.

1. **Open** `https://hydrax-context-production.up.railway.app/`
   The cover page. 60 seconds reading time. Anchors the framing.
2. **Click** "View 9-Slide Deck" → `/deck`
   14-slide deck, walked top-to-bottom or jumped via slide nav. Stop at slide 8 to mirror the narrated script, or push through 9-13 as bonus.
3. **Click** "View Source" → GitHub. 30-second tour: top-level README, `services/`, `web/apps/`, `docs/plans/`. Demonstrates breadth and discipline.
4. **Open in new tab** `https://hydrax-portals-production.up.railway.app/`
   Marketing landing. Visually polished. Walk the 5-portal grid + the solutions menu.
5. **DO NOT click any "Try" CTA.** Instead, manually type `/issuer/` (trailing slash, home route only). It renders the issuer hero + EmptyState. Repeat for `/distributor/`, `/investor/`, `/ops/`, `/admin/` if time permits.
6. **Close on** `https://hydrax-context-production.up.railway.app/script` — shows you've thought about how to narrate this end-to-end in five minutes.

**One-line interviewer recommendation that won't backfire:**
> "Two URLs to look at if you want — `hydrax-context-production.up.railway.app` is the architecture write-up and deck; `hydrax-portals-production.up.railway.app` is the live multi-portal shell. The portal data layer is the next ship — see the deferrals section in the deck for why."

That last clause inoculates against a "Try" click. If they click anyway and hit the error, they've already heard the framing — it lands as "yeah, he flagged that" not "his demo is broken".

## My recommendation

**Path A + Path Z**, in that order, before the demo. Path A is 15 minutes and removes the only impression-killer. Path Z is the click-path you actually walk in the meeting. Path B is the right long-term move but is a bad bet on demo eve unless we have real buffer.

### Path B — replaced with Path B-demo, landed 2026-04-27 (commit 1938166)

Investigation surfaced two facts that changed the calculus:
1. **Auth gate.** BFF `/v1/products` requires `requireSession`. The "minimum viable real BFF" is BFF + integration-svc + workflow-svc + Postgres + magic-link SMTP. ~3-6h realistic, first-deploy failure probability >50% per service.
2. **Demo-mode flag already exists.** `web/packages/api-client/src/api.ts:251` — `const baseQuery = DEMO_MODE ? demoBaseQuery : realBaseQuery;`. Setting `VITE_DEMO_MODE=true` at build time activates the entire fixtures path: 3 institutional products (SDC-2026-Q2-A, MMF-2026-USD-1, TBILL-2026-Q3 — matching the PRD product wedge), audit events, approvals, transitions, composite health.

**Executed:** rebuilt all 5 portal SPAs with `VITE_DEMO_MODE=true VITE_BASE_PATH=/<role>/`, synced `dist/` → `web/portal-deploy/<role>/`, deployed to hydrax-portals. Production verified — every bundle has `VITE_DEMO_MODE:"true"` baked into the Vite env object plus the fixture strings (`SDC-2026-Q2-A`, `MMF-2026-USD-1`, `Treasury Bill`). All 21 portal routes return 200 (including data routes like `/issuer/products`, `/admin/composability`, `/investor/health`, `/ops/audit`).

**Trade-off accepted:** mutations do NOT round-trip (createProduct, transitionProduct, decideApproval return synthesized response shapes but the next query returns the original list). For the interview demo this is fine — the flow is read-mostly. For a "this is production" pitch, the next ship is the real BFF chain. Honest framing in the deck remains intact: "synthetic data; backend chain is the next ship — see deferrals slide".
